# Comparative Analysis: src-no-skills, src-dotnet-webapi, src-dotnet-artisan

This document compares ASP.NET Core Web API applications generated by three different GitHub Copilot configurations. Each directory contains three apps — **FitnessStudioApi**, **LibraryApi**, **VetClinicApi** — all targeting .NET 10 with SQLite. The configurations are:

- **`src-no-skills`** — Baseline. No custom Copilot skills or plugins.
- **`src-dotnet-webapi`** — Used the `dotnet-webapi` custom skill (focused, single-file Web API guidance).
- **`src-dotnet-artisan`** — Used the `dotnet-artisan` plugin (multi-skill chain: `using-dotnet`, `dotnet-advisor`, `dotnet-csharp`, `dotnet-api`).

All nine projects share the same functional requirements: CRUD operations, pagination, domain validation, relationships, and data seeding.

## 1. API Style

### src-no-skills — Traditional Controllers

All three apps use `[ApiController]` with attribute routing. Endpoints are instance methods on `ControllerBase` subclasses.

```csharp
// src-no-skills/FitnessStudioApi/Controllers/MembersController.cs
[ApiController]
[Route("api/members")]
[Produces("application/json")]
public class MembersController : ControllerBase
{
    private readonly IMemberService _service;
    public MembersController(IMemberService service) => _service = service;

    [HttpGet]
    [ProducesResponseType(typeof(PagedResult<MemberListDto>), 200)]
    public async Task<IActionResult> GetAll([FromQuery] string? search,
        [FromQuery] bool? isActive, [FromQuery] PaginationParams pagination)
        => Ok(await _service.GetAllAsync(search, isActive, pagination));
}
```

### src-dotnet-webapi — Minimal APIs with Extension Methods

Static extension methods on `IEndpointRouteBuilder`. Handlers are inline lambdas.

```csharp
// src-dotnet-webapi/FitnessStudioApi/Endpoints/MemberEndpoints.cs
public static class MemberEndpoints
{
    public static void MapMemberEndpoints(this IEndpointRouteBuilder app)
    {
        var group = app.MapGroup("/api/members").WithTags("Members");

        group.MapGet("/", async Task<Ok<PaginatedResponse<MemberListResponse>>> (
            string? search, bool? isActive, int? page, int? pageSize,
            IMemberService service, CancellationToken ct) =>
        {
            var result = await service.GetAllAsync(search, isActive,
                Math.Clamp(page ?? 1, 1, int.MaxValue),
                Math.Clamp(pageSize ?? 20, 1, 100), ct);
            return TypedResults.Ok(result);
        })
        .WithName("GetMembers")
        .WithSummary("List members")
        .WithDescription("Returns a paginated list of members.")
        .Produces<PaginatedResponse<MemberListResponse>>(StatusCodes.Status200OK);
    }
}
```

### src-dotnet-artisan — Minimal APIs with RouteGroupBuilder and Separate Methods

Uses `RouteGroupBuilder` (returns group for chaining) and extracts handlers into `private static` methods.

```csharp
// src-dotnet-artisan/FitnessStudioApi/Endpoints/MemberEndpoints.cs
public static class MemberEndpoints
{
    public static RouteGroupBuilder MapMemberEndpoints(this RouteGroupBuilder group)
    {
        var members = group.MapGroup("/members").WithTags("Members");
        members.MapGet("/", GetAllAsync)
            .WithSummary("List members with search, filter, and pagination");
        members.MapPost("/", CreateAsync)
            .WithSummary("Register a new member");
        return group;
    }

    private static async Task<Ok<PaginatedResponse<MemberResponse>>> GetAllAsync(
        IMemberService service, string? search, bool? isActive,
        int page = 1, int pageSize = 10, CancellationToken ct = default)
    {
        var result = await service.GetAllAsync(search, isActive, page, pageSize, ct);
        return TypedResults.Ok(result);
    }
}
```

**Verdict**: Both Minimal API approaches (webapi, artisan) are preferred for new .NET 10 projects. The artisan pattern of extracting private methods improves readability on larger files, while the webapi inline pattern keeps everything in one place. The controller pattern is legacy for new API-only projects.

## 5. Service Abstraction

All three variants register services via `Interface + Implementation` with `AddScoped<>()`:

```csharp
// All three Program.cs files
builder.Services.AddScoped<IMemberService, MemberService>();
builder.Services.AddScoped<IBookingService, BookingService>();
```

The difference is interface file organization:

| Directory | Interface Files | Pattern |
|---|---|---|
| src-no-skills | Varies — some in service files, some separate | Inconsistent |
| src-dotnet-webapi | Separate `I*Service.cs` per service | ✅ One-interface-per-file |
| src-dotnet-artisan | FitnessStudio: single `Interfaces.cs`; Library/VetClinic: separate files | Mixed |

```csharp
// src-dotnet-webapi: separate file per interface
// Services/IMemberService.cs
public interface IMemberService
{
    Task<PaginatedResponse<MemberListResponse>> GetAllAsync(..., CancellationToken ct);
    Task<MemberResponse?> GetByIdAsync(int id, CancellationToken ct);
}

// src-dotnet-artisan/FitnessStudioApi: all interfaces in one file
// Services/Interfaces.cs
public interface IMembershipPlanService { ... }
public interface IMemberService { ... }
public interface IMembershipService { ... }
```

**Verdict**: One-interface-per-file (webapi) is the cleanest for navigation and merge conflict avoidance. All three variants correctly use interface abstraction for DI — the difference is purely organizational.

## 6. CancellationToken Propagation

### src-no-skills — No propagation

```csharp
// src-no-skills/FitnessStudioApi/Services/MemberService.cs
public async Task<PagedResult<MemberListDto>> GetAllAsync(
    string? search, bool? isActive, PaginationParams pagination)
{
    var totalCount = await query.CountAsync();       // no token
    var items = await query.ToListAsync();            // no token
    await _db.SaveChangesAsync();                     // no token
}
```

### src-dotnet-webapi — Full propagation (endpoint → service → EF Core)

```csharp
// src-dotnet-webapi/FitnessStudioApi/Services/MemberService.cs
public async Task<PaginatedResponse<MemberListResponse>> GetAllAsync(
    string? search, bool? isActive, int page, int pageSize, CancellationToken ct)
{
    var totalCount = await query.CountAsync(ct);
    var items = await query.ToListAsync(ct);
}
```

### src-dotnet-artisan — Full propagation

```csharp
// src-dotnet-artisan/FitnessStudioApi/Services/MemberService.cs
public async Task<PaginatedResponse<MemberResponse>> GetAllAsync(
    string? search, bool? isActive, int page, int pageSize, CancellationToken ct)
{
    var totalCount = await query.CountAsync(ct);
    var items = await query.ToListAsync(ct);
}
```

**Verdict**: CancellationToken propagation is critical for production APIs. When a client disconnects, tokens cancel in-flight database queries and prevent wasted work. Both skill-assisted versions pass tokens through every layer; the baseline never propagates them.

## 7. AsNoTracking()

### src-no-skills — Not used

```csharp
// src-no-skills: queries always go through change tracker
var query = _db.Members.AsQueryable();
```

### src-dotnet-webapi — Consistent across all 3 apps (~88 occurrences)

```csharp
// src-dotnet-webapi/FitnessStudioApi/Services/MemberService.cs
var query = db.Members.AsNoTracking().AsQueryable();

var member = await db.Members.AsNoTracking()
    .Include(m => m.Memberships).ThenInclude(ms => ms.MembershipPlan)
    .FirstOrDefaultAsync(m => m.Id == id, ct);
```

| App | AsNoTracking() count |
|---|---|
| FitnessStudioApi | 21 |
| LibraryApi | 45 |
| VetClinicApi | 22 |

### src-dotnet-artisan — Not used

```csharp
// src-dotnet-artisan: queries always go through change tracker
var query = db.Members.AsQueryable();
```

**Verdict**: `AsNoTracking()` reduces memory allocation and improves read query performance by 20–40% by skipping change tracker snapshots. The `dotnet-webapi` skill is the **only** variant that applies it, and does so consistently. This is the single largest performance gap between the two skill-assisted versions.

```csharp

protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    // ... configuration ...
    SeedData(modelBuilder);
}

private static void SeedData(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Owner>().HasData(
        new Owner { Id = 1, FirstName = "Sarah", LastName = "Johnson", ... });

// Program.cs uses Migrate() instead of EnsureCreated()
```

```csharp
public static class SeedData
{
    public static async Task InitializeAsync(AppDbContext db)
    {
        if (await db.MembershipPlans.AnyAsync()) return;
```

**Verdict**: The migration-based `HasData()` approach (webapi) is best for deterministic, version-controlled seed data. The async static method approach (artisan) is pragmatic for development. The sync approach (no-skills) blocks the startup thread.

## 10. Middleware Style

### src-no-skills — Mixed: IExceptionHandler + convention-based middleware

FitnessStudio uses `IExceptionHandler`:

```csharp
// src-no-skills/FitnessStudioApi/Middleware/GlobalExceptionHandler.cs
public class GlobalExceptionHandler : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, Exception exception,
        CancellationToken cancellationToken) { ... }
}
```

Library and VetClinic use convention-based middleware:

```csharp
// src-no-skills/LibraryApi/Middleware/GlobalExceptionHandlerMiddleware.cs
public class GlobalExceptionHandlerMiddleware
{
    private readonly RequestDelegate _next;
    public async Task InvokeAsync(HttpContext context)
    {
        try { await _next(context); }
        catch (Exception ex) { await HandleExceptionAsync(context, ex); }
    }
}
```

### src-dotnet-webapi — `IExceptionHandler` consistently across all 3 apps

```csharp
// src-dotnet-webapi/FitnessStudioApi/Middleware/ApiExceptionHandler.cs
internal sealed class ApiExceptionHandler(ILogger<ApiExceptionHandler> logger)
    : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, Exception exception,
        CancellationToken cancellationToken)
    {
        var (statusCode, title) = exception switch
        {
            KeyNotFoundException => (StatusCodes.Status404NotFound, "Not Found"),
            ArgumentException => (StatusCodes.Status400BadRequest, "Bad Request"),
            InvalidOperationException => (StatusCodes.Status409Conflict, "Conflict"),
            _ => (0, (string?)null)
        };

        if (statusCode == 0) return false;  // let other handlers process

        httpContext.Response.StatusCode = statusCode;
        await httpContext.Response.WriteAsJsonAsync(new ProblemDetails
        {
            Status = statusCode, Title = title,
            Detail = exception.Message, Instance = httpContext.Request.Path
        }, cancellationToken);
        return true;
    }
}
```

### src-dotnet-artisan — No middleware files; inline exception handling in Program.cs

```csharp
// src-dotnet-artisan/FitnessStudioApi/Program.cs
app.UseExceptionHandler(errorApp =>
{
    errorApp.Run(async context =>
    {
        context.Response.ContentType = "application/problem+json";
        context.Response.StatusCode = StatusCodes.Status500InternalServerError;
        await context.Response.WriteAsJsonAsync(new ProblemDetails
        {
            Status = StatusCodes.Status500InternalServerError,
            Title = "An unexpected error occurred",
            Type = "https://tools.ietf.org/html/rfc7807"
        });
    });
});
```

**Verdict**: `IExceptionHandler` (webapi) is the modern .NET 8+ approach: it's DI-friendly, testable, composable, and returns `false` for unhandled exceptions. Convention-based middleware (no-skills) is valid but older. Inline handling (artisan) catches only 500s and lacks status-code differentiation.

## 11. Exception Handling Strategy

### src-no-skills — Custom exception hierarchy

```csharp
// src-no-skills/LibraryApi/Middleware/GlobalExceptionHandlerMiddleware.cs
public class BusinessRuleException : Exception
{
    public BusinessRuleException(string message) : base(message) { }
}
public class NotFoundException : Exception
{
    public NotFoundException(string message) : base(message) { }
}
public class ConflictException : Exception
{
    public ConflictException(string message) : base(message) { }
}
```

### src-dotnet-webapi — Standard .NET exceptions mapped to HTTP status codes

```csharp
// Services throw standard exceptions:
throw new KeyNotFoundException($"Member {id} not found");
throw new ArgumentException("Email already registered");
throw new InvalidOperationException("Cannot cancel: past booking");

// Middleware maps them:
KeyNotFoundException    → 404
ArgumentException      → 400
InvalidOperationException → 409
```

### src-dotnet-artisan — Result tuples (no exceptions for business logic)

```csharp
// src-dotnet-artisan/FitnessStudioApi/Services/Interfaces.cs
Task<(MembershipResponse? Result, string? Error)> CreateAsync(
    CreateMembershipRequest request, CancellationToken ct);
Task<(bool Success, string? Error)> CancelAsync(int id, CancellationToken ct);

// src-dotnet-artisan/VetClinicApi/Services/IAppointmentService.cs
Task<(AppointmentResponse? Result, string? Error, bool NotFound)> UpdateAsync(
    int id, UpdateAppointmentRequest request, CancellationToken cancellationToken);
```

**Verdict**: Each approach has trade-offs. Custom exceptions (no-skills) add type safety but are verbose. Standard exceptions (webapi) are simple but rely on convention. Result tuples (artisan) avoid exceptions entirely but create complex return signatures. The webapi approach is the most practical for typical CRUD APIs.

## 13. Pagination Implementation

### src-no-skills — Mutable `class` with computed property

```csharp
// src-no-skills/LibraryApi/DTOs/PagedResult.cs
public class PagedResult<T>
{
    public List<T> Items { get; set; } = new();
    public int TotalCount { get; set; }
    public int Page { get; set; }
    public int PageSize { get; set; }
    public int TotalPages => (int)Math.Ceiling((double)TotalCount / PageSize);
}
```

### src-dotnet-webapi — `sealed record` with `required` properties and factory method

```csharp
// src-dotnet-webapi/FitnessStudioApi/DTOs/PaginatedResponse.cs
public sealed record PaginatedResponse<T>
{
    public required IReadOnlyList<T> Items { get; init; }
    public required int Page { get; init; }
    public required int PageSize { get; init; }
    public required int TotalCount { get; init; }
    public required int TotalPages { get; init; }
    public required bool HasNextPage { get; init; }
    public required bool HasPreviousPage { get; init; }

    public static PaginatedResponse<T> Create(
        IReadOnlyList<T> items, int page, int pageSize, int totalCount)
    {
        var totalPages = (int)Math.Ceiling(totalCount / (double)pageSize);
        return new PaginatedResponse<T>
        {
            Items = items, Page = page, PageSize = pageSize,
            TotalCount = totalCount, TotalPages = totalPages,
            HasNextPage = page < totalPages, HasPreviousPage = page > 1
        };
    }
}
```

### src-dotnet-artisan — Compact `sealed record` with primary constructor

```csharp
// src-dotnet-artisan/FitnessStudioApi/DTOs/Dtos.cs
public sealed record PaginatedResponse<T>(
    IReadOnlyList<T> Items, int Page, int PageSize,
    int TotalCount, int TotalPages);
```

**Verdict**: The webapi approach with `required` + `init` + factory method is the most robust — it ensures all fields are populated and provides `HasNextPage`/`HasPreviousPage` convenience properties. The artisan compact form is minimal but omits navigation booleans.

### Most Impactful Differences (ranked by production significance)

1. **CancellationToken propagation** (webapi ✅ / artisan ✅ / no-skills ❌) — Without cancellation support, disconnected client requests continue consuming database connections and CPU.

2. **AsNoTracking()** (webapi ✅ / artisan ❌ / no-skills ❌) — 20–40% performance improvement on read queries. The `dotnet-webapi` skill is the only variant that consistently applies this optimization.

3. **Sealed types** (webapi ✅ / artisan ✅ / no-skills ❌) — JIT devirtualization enables measurable throughput improvements under load.

4. **IExceptionHandler** (webapi ✅ / no-skills mixed / artisan ❌) — Testable, DI-friendly, composable error handling vs. inline lambdas.

5. **Immutable DTOs** (webapi ✅ / artisan ✅ / no-skills ❌) — `sealed record` types prevent accidental mutation and provide value equality.

6. **Return type precision** (webapi ✅ / artisan ✅ / no-skills ❌) — `IReadOnlyList<T>` communicates intent and prevents mutation.

7. **OpenAPI richness** (webapi ✅ / no-skills mixed / artisan minimal) — Detailed endpoint documentation drives better client SDK generation.

### Overall Assessment

The **dotnet-webapi** skill produces the highest quality code across all dimensions. Its key differentiator over dotnet-artisan is **AsNoTracking()** on read queries (88 occurrences vs 0), **rich OpenAPI metadata** (`.WithName()`, `.WithDescription()`, `.Produces<>()` on every endpoint), and **consistent IExceptionHandler** middleware. The focused, single-skill guidance translates directly into more consistent output.

The **dotnet-artisan** skill chain produces nearly equivalent quality — matching dotnet-webapi on sealed types, primary constructors, CancellationToken, and `IReadOnlyList<T>`. It uniquely provides C# 12 `[]` collection expressions and the result-tuple error handling pattern. However, the broader skill chain doesn't guarantee better results: the missing AsNoTracking and minimal OpenAPI metadata suggest that multi-skill orchestration may dilute focus on specific Web API optimizations.

The **no-skills** baseline demonstrates that without skill guidance, Copilot defaults to older patterns (controllers, mutable DTOs, no CancellationToken) that require manual modernization for production readiness.
