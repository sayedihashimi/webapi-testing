# Comparative Analysis: no-skills, dotnet-webapi, dotnet-artisan, managedcode-dotnet-skills, dotnet-skills

## Introduction

This report analyzes three ASP.NET Razor Pages applications generated by GitHub Copilot under five different skill configurations. Each configuration produced the same three apps:

| App | Domain |
|-----|--------|
| **SparkEvents** | Event registration portal with ticket types, capacity management, waitlists, and check-in |
| **KeystoneProperties** | Property management with leases, tenants, maintenance requests, and rent tracking |
| **HorizonHR** | Employee directory and HR portal with leave management, reviews, and skills |

**Configurations tested:**

| Directory | Configuration | Description |
|-----------|--------------|-------------|
| `no-skills` | Baseline | Default Copilot with no custom skills |
| `dotnet-webapi` | dotnet-webapi skill | Single .NET web API skill |
| `dotnet-artisan` | dotnet-artisan plugin | Plugin chain with 9 skills and 14 specialist agents |
| `managedcode-dotnet-skills` | Community managed-code skills | Community-contributed .NET skills |
| `dotnet-skills` | Official .NET Skills (dotnet/skills) | Official .NET team skills |

All apps target **.NET 10**, use **SQLite** with **EF Core**, and have **Nullable Reference Types** and **Implicit Usings** enabled.

## 1. Page Model Design

All five configurations produce **thin page models** that delegate business logic to injected services. No configuration injects `DbContext` directly into page models.

```csharp
// no-skills — traditional constructor with private fields
public class CreateModel : PageModel
{
    private readonly IEventService _eventService;
    private readonly ICategoryService _categoryService;
    public CreateModel(IEventService eventService, ICategoryService categoryService)
    {
        _eventService = eventService;
        _categoryService = categoryService;
    }
}
```

```csharp
// dotnet-artisan — primary constructor, sealed class
public sealed class CreateModel(IEventService eventService, SparkEventsDbContext db) : PageModel
{
    // Uses primary constructor parameters directly — no private fields
}
```

```csharp
// dotnet-skills — traditional constructor, not sealed
public class EventService : IEventService
{
    private readonly SparkEventsDbContext _db;
    private readonly ILogger<EventService> _logger;
    public EventService(SparkEventsDbContext db, ILogger<EventService> logger)
    {
        _db = db;
        _logger = logger;
    }
}
```

**Verdict:** All configurations get this right. The dotnet-artisan and dotnet-webapi versions are slightly better due to sealed classes and primary constructors reducing boilerplate.

## 2. Form Handling & Validation

All configurations implement the core form handling patterns: `[BindProperty]` with InputModel classes, `asp-validation-summary`, `asp-validation-for`, and the PRG pattern.

| Metric | no-skills | dotnet-webapi | dotnet-artisan | managedcode | dotnet-skills |
|--------|-----------|---------------|----------------|-------------|---------------|
| `asp-validation-summary` | 37 | 35 | 42 | 37 | 39 |
| `_ValidationScriptsPartial` | 16 | 28 | 29 | 29 | 41 |
| `RedirectToPage` (PRG) | 62 | 73 | 66 | 68 | 75 |

Notable: **dotnet-artisan** uses `IValidatableObject` for cross-field validation:

```csharp
// dotnet-artisan — Events/Create.cshtml.cs
public sealed class InputModel : IValidatableObject
{
    [Required, MaxLength(300)]
    public string Title { get; set; } = string.Empty;

    public IEnumerable<ValidationResult> Validate(ValidationContext validationContext)
    {
        if (EndDate <= StartDate)
            yield return new ValidationResult(
                "End date must be after start date.", [nameof(EndDate)]);
    }
}
```

**no-skills** performs the same validation inline in the handler, which is less reusable:

```csharp
// no-skills — Events/Create.cshtml.cs
if (Input.EndDate <= Input.StartDate)
{
    ModelState.AddModelError("Input.EndDate", "End date must be after start date.");
    await LoadSelectListsAsync();
    return Page();
}
```

**Verdict:** dotnet-artisan's `IValidatableObject` approach is best practice — it co-locates validation with the model. dotnet-skills has the highest `_ValidationScriptsPartial` usage (41), suggesting more consistent client-side validation inclusion.

## 4. Named Handler Methods

All configurations use roughly the same number of named POST handlers (~52-53). The key difference is in how complex actions are structured.

```csharp
// dotnet-artisan — separate pages for each action
Pages/Events/
├── Cancel.cshtml.cs    // OnPostAsync() for cancel
├── Complete.cshtml.cs  // OnPostAsync() for complete
└── Register.cshtml.cs  // OnPostAsync() for register
```

```csharp
// dotnet-webapi — named handlers on Details page
public async Task<IActionResult> OnPostPublishAsync(int id, CancellationToken ct) { ... }
public async Task<IActionResult> OnPostCancelAsync(int id, string reason, CancellationToken ct) { ... }
public async Task<IActionResult> OnPostCompleteAsync(int id, CancellationToken ct) { ... }
```

**Verdict:** Both approaches are valid Razor Pages patterns. The separate-page approach (dotnet-artisan, dotnet-skills) has better separation of concerns. The named-handler approach (dotnet-webapi) reduces file count but creates larger page models.

### Most Impactful Differences (Production Readiness)

1. **CancellationToken Propagation** (dotnet-webapi only) — **Critical** for production. Prevents wasted server resources on cancelled requests. The 423 vs 6 gap is enormous.

2. **AsNoTracking Usage** (all skills, not baseline) — **High impact** for read-heavy apps. no-skills' complete omission means every read query unnecessarily tracks entities, doubling memory usage.

3. **Sealed Types** (dotnet-webapi best) — **Medium impact**. Enables JIT devirtualization optimizations and signals design intent. The 187 vs 0 gap shows skill-guided code quality.

4. **Exception Handling Infrastructure** (dotnet-webapi only) — **High impact** for production. Custom middleware with structured error logging and status code mapping. All other configurations rely on default error pages.

5. **Accessibility** (managedcode best, all skills > baseline) — **High impact** for compliance. The 112 vs 19 ARIA attribute gap means skills produce significantly more accessible applications.

6. **Primary Constructors** (dotnet-artisan, managedcode best) — **Medium impact** for developer productivity. Eliminates ~3-5 lines of boilerplate per class.

### Overall Assessment

**Skills make a measurable difference.** The baseline (no-skills) configuration produces functional code that follows many best practices but misses critical production concerns: `AsNoTracking`, `CancellationToken`, `sealed` types, comprehensive ARIA attributes, and modern C# features.

**dotnet-webapi** produces the most production-ready code overall, with the strongest performance optimizations and exception handling. **dotnet-artisan** produces the most modern C# with the best developer ergonomics. **managedcode** leads in accessibility and UI components. **dotnet-skills** excels in logging and Bootstrap UI polish.

No single configuration produces a "perfect" result — the ideal app would combine dotnet-webapi's performance patterns, dotnet-artisan's modern C# syntax, managedcode's accessibility, and dotnet-skills' logging depth.
