## 3. Input Validation & Guard Clauses [CRITICAL]

All configurations use data annotation validation attributes (`[Required]`, `[MaxLength]`, `[Range]`, `[EmailAddress]`) on model and InputModel classes. All check `ModelState.IsValid` in POST handlers.

**dotnet-artisan and dotnet-webapi** go further with service-level guard clauses:

```csharp
// dotnet-artisan — RegistrationService.cs
var evt = await db.Events.FirstOrDefaultAsync(e => e.Id == eventId)
    ?? throw new InvalidOperationException("Event not found.");

if (evt.Status is EventStatus.Completed or EventStatus.Cancelled)
    throw new InvalidOperationException("Cannot cancel a completed/cancelled event.");
```

```csharp
// dotnet-webapi — EventService.cs
var evt = await db.Events.Include(e => e.TicketTypes)
    .FirstOrDefaultAsync(e => e.Id == id, ct);
if (evt is null) return false;

if (evt.Status != EventStatus.Draft)
    throw new InvalidOperationException("Only draft events can be published.");
```

**no-skills** uses simpler guard patterns:

```csharp
// no-skills — EventService.cs
if (evt == null || evt.Status != EventStatus.Draft) return false;
```

**dotnet-artisan** also uses `IValidatableObject` for cross-field validation:

```csharp
// dotnet-artisan — Events/Create.cshtml.cs InputModel
public sealed class InputModel : IValidatableObject
{
    public IEnumerable<ValidationResult> Validate(ValidationContext ctx)
    {
        if (EndDate <= StartDate)
            yield return new ValidationResult("End date must be after start date.",
                [nameof(EndDate)]);
    }
}
```

**Scores:**
- **dotnet-artisan: 5** — IValidatableObject, comprehensive guard clauses, throw expressions
- **dotnet-webapi: 5** — Comprehensive guard clauses, business rule validation in services
- **no-skills: 4** — Good validation attributes, basic guard patterns, no IValidatableObject
- **dotnet-skills: 4** — Good validation, some guard clauses
- **managedcode: 4** — Good validation, some guard clauses

**Verdict:** dotnet-artisan and dotnet-webapi lead with defense-in-depth validation at both form and service layers.

## 5. EF Migration Usage [CRITICAL]

**Every configuration uses `EnsureCreated()`/`EnsureCreatedAsync()` instead of EF Core migrations.** No project calls `Migrate()` or `MigrateAsync()`, and no migrations folder exists in any project.

```csharp
// Pattern used by ALL configs
using (var scope = app.Services.CreateScope())
{
    var db = scope.ServiceProvider.GetRequiredService<SparkEventsDbContext>();
    await db.Database.EnsureCreatedAsync();

}
```

**Scores:** All configurations: **2** — EnsureCreated bypasses migrations, making schema evolution impossible. While understandable for a SQLite demo, the prompt specified the app should use EF Core best practices.

**Verdict:** Universal anti-pattern. No configuration gets this right.

## Summary: Impact of Skills

### Most Impactful Differences (ranked by score delta)

1. **Error Handling Strategy** (Δ = 2.0) — dotnet-webapi is the only configuration with custom exception handling middleware (`GlobalExceptionHandlerMiddleware`, `IExceptionHandler`). This is a production-critical pattern that no other configuration generates.

2. **Modern C# Adoption** (Δ = 2.0) — dotnet-artisan and dotnet-webapi produce C# 12 code (primary constructors, collection expressions, sealed classes) while no-skills and dotnet-skills generate C# 10-era patterns. This is the most visible quality signal.

3. **Async Patterns & Cancellation** (Δ = 3.0) — dotnet-webapi is the only configuration that consistently propagates `CancellationToken` from page handlers through service layers to EF Core calls. no-skills has zero CancellationToken usage.

4. **NuGet Package Discipline** (Δ = 2.0) — dotnet-artisan is the only configuration that pins exact package versions (10.0.5). All others use wildcard patterns that risk non-reproducible builds.

5. **EF Core Best Practices** (Δ = 2.0) — no-skills is the only configuration that completely omits `AsNoTracking()` for read queries, missing a ~2x performance improvement on read paths.

### Overall Assessment

**dotnet-webapi (183.5 points)** produces the most production-ready code overall. Its strengths are comprehensive CancellationToken propagation, custom exception middleware, HSTS configuration, and modern C# patterns. Its only weakness is wildcard package versions.

**dotnet-artisan (170.5 points)** generates the highest-quality code patterns (sealed classes, exact package versions, primary constructors) but is critically penalized by the HorizonHR build failure and missing HSTS/HTTPS. If all apps built successfully, it would likely score first.

**managedcode-dotnet-skills (147 points)** and **dotnet-skills (146 points)** produce good code that improves on the baseline in some areas (AsNoTracking, some CancellationToken) but don't match the top-tier configurations for modern C# adoption.

**no-skills (141 points)** establishes a solid baseline — the code is correct, well-organized, and follows core ASP.NET conventions. However, it misses performance optimizations (AsNoTracking), modern language features (primary constructors, sealed), and advanced patterns (CancellationToken, exception middleware) that the skill-enhanced configurations provide.

**The bottom line:** Skills demonstrably improve code quality. The dotnet-webapi skill produces the most complete and production-ready output, while dotnet-artisan generates the most stylistically modern code. Both significantly outperform the baseline in weighted scoring (+42.5 and +29.5 points respectively).
