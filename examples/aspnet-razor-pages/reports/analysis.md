# Comparative Analysis: no-skills, dotnet-webapi, dotnet-artisan, managedcode-dotnet-skills

## Introduction

This report compares ASP.NET Core Razor Pages applications generated by **four different Copilot skill configurations**. Each configuration produced three identical application scenarios:

| Configuration | Description |
|---|---|
| **no-skills** | Baseline — no custom skills attached |
| **dotnet-webapi** | Single `dotnet-webapi` skill (API-focused .NET guidance) |
| **dotnet-artisan** | Multi-skill plugin chain (`using-dotnet` → `dotnet-advisor` → `dotnet-csharp` → `dotnet-api`) |
| **managedcode-dotnet-skills** | Community skills (`dotnet-aspnet-core`, `dotnet-entity-framework-core`, `dotnet-modern-csharp`, `dotnet-project-setup`) |

Each configuration generated three apps targeting **.NET 10** with **SQLite** and **Bootstrap 5**:

- **SparkEvents** — Event registration portal with ticket types, capacity management, waitlists, and check-in
- **KeystoneProperties** — Property management with leases, tenants, maintenance requests, and rent tracking
- **HorizonHR** — Employee directory and HR portal with leave management, reviews, and skills

> **Note:** A fifth configuration (`dotnet-skills` — Official .NET Skills from dotnet/skills) was listed as expected but no output directory was present.

## 1. Page Model Design

Page model thickness is the most significant architectural difference across configurations.

**no-skills** — Fat page models inject `DbContext` directly alongside services:

```csharp
// no-skills/SparkEvents/Pages/Events/Index.cshtml.cs
public class IndexModel : PageModel
{
    private readonly IEventService _eventService;
    private readonly SparkEventsDbContext _db;  // DbContext injected directly

    public IndexModel(IEventService eventService, SparkEventsDbContext db)
    {
        _eventService = eventService;
        _db = db;
    }

    public async Task OnGetAsync(...)
    {
        Categories = await _db.EventCategories.OrderBy(c => c.Name).ToListAsync();
    }
}
```

**dotnet-artisan** — Hybrid approach: services for business logic, DbContext for simple lookups:

```csharp
// dotnet-artisan/SparkEvents/Pages/Events/Register.cshtml.cs
public sealed class RegisterModel(
    SparkEventsDbContext db,
    IRegistrationService registrationService) : PageModel
{
    public async Task<IActionResult> OnPostAsync(int eventId)
    {
        var evt = await db.Events.AsNoTracking()  // DbContext for read
            .FirstOrDefaultAsync(e => e.Id == eventId);
        var registration = await registrationService.RegisterAsync(...);  // Service for logic
    }
}
```

**dotnet-webapi** — Thin models with services only; no DbContext in pages:

```csharp
// dotnet-webapi/SparkEvents/Pages/Events/Index.cshtml.cs
public sealed class IndexModel : PageModel
{
    private readonly IEventService _eventService;
    private readonly IEventCategoryService _categoryService;  // All via services

    public async Task OnGetAsync(CancellationToken ct)
    {
        var categories = await _categoryService.GetAllForDropdownAsync(ct);
        Events = await _eventService.GetAllAsync(Search, CategoryId, Status, ...ct);
    }
}
```

**managedcode-dotnet-skills** — Also thin, with primary constructor syntax:

```csharp
// managedcode-dotnet-skills/SparkEvents/Pages/Attendees/Create.cshtml.cs
public class CreateModel(IAttendeeService attendeeService) : PageModel
{
    public async Task<IActionResult> OnPostAsync()
    {
        var existing = await attendeeService.GetByEmailAsync(Input.Email);
        // All access through service interface
    }
}
```

**Verdict:** **dotnet-webapi** and **managedcode-dotnet-skills** follow best practices — thin page models with full service delegation. This makes page models testable without requiring a database. The dotnet-artisan hybrid approach (KISS: DbContext for simple reads) is pragmatic but introduces inconsistency.

## 2. Sealed Types

A stark binary divide: two configurations seal everything, two seal nothing.

```csharp
// dotnet-artisan — sealed on models, services, AND page models
public sealed class Event { ... }
public sealed class EventService(SparkEventsDbContext db, ILogger<EventService> logger) : IEventService { ... }
public sealed class IndexModel(IEventService eventService) : PageModel { ... }

// no-skills — no sealed modifier anywhere
public class Event { ... }
public class EventService : IEventService { ... }
public class IndexModel : PageModel { ... }
```

| Config | Sealed Classes | Non-Sealed | % Sealed |
|---|---|---|---|
| **no-skills** | 0 | ~176 | 0% |
| **dotnet-artisan** | ~179 | ~6 | 97% |
| **dotnet-webapi** | ~188 | ~3 | 98% |
| **managedcode-dotnet-skills** | 0 | ~199 | 0% |

**Verdict:** **dotnet-artisan** and **dotnet-webapi** are correct. Sealing classes not designed for inheritance follows CA1852, communicates design intent, and enables JIT devirtualization. The `managedcode-dotnet-skills` configuration's omission is a notable gap given its use of other modern patterns.

## 4. CancellationToken Propagation

Only **dotnet-webapi** propagates cancellation tokens through the entire call chain.

```csharp
// dotnet-webapi — CancellationToken in page handler, forwarded to service, forwarded to EF Core
public async Task OnGetAsync(CancellationToken ct)
{
    Events = await _eventService.GetAllAsync(Search, CategoryId, Status, ..., ct);
}

// Service layer
public async Task<PaginatedList<Event>> GetAllAsync(..., CancellationToken ct = default)
{
    var totalCount = await query.CountAsync(ct);
    var items = await query.Skip(...).Take(...).ToListAsync(ct);
}
```

```csharp
// no-skills / dotnet-artisan / managedcode — no CancellationToken at all
public async Task OnGetAsync(string? search, int? categoryId, ...)
{
    var (items, totalCount) = await _eventService.GetEventsAsync(search, categoryId, ...);
    // No way to cancel if user navigates away
}
```

| Config | Files with CancellationToken | Pattern |
|---|---|---|
| **no-skills** | 0 page handlers | Not used |
| **dotnet-artisan** | 0 page handlers | Not used |
| **dotnet-webapi** | ~100+ files | Pervasive — handlers → services → EF Core |
| **managedcode-dotnet-skills** | 1 (DbContext override) | Minimal |

**Verdict:** **dotnet-webapi** is the clear winner. CancellationToken propagation is critical for production — it prevents wasted server resources when users navigate away or connections drop. The absence in other configs is the single largest production-readiness gap.

## 5. AsNoTracking Usage

Read-only query optimization shows dramatic differences.

```csharp
// dotnet-artisan / dotnet-webapi / managedcode — AsNoTracking on read queries
var query = db.Events
    .AsNoTracking()
    .Include(e => e.EventCategory)
    .Include(e => e.Venue)
    .AsQueryable();

// no-skills — missing AsNoTracking (tracking overhead on every read)
var query = _db.Events
    .Include(e => e.EventCategory)
    .Include(e => e.Venue)
    .AsQueryable();
```

| Config | Total AsNoTracking Calls | Coverage |
|---|---|---|
| **no-skills** | 2 | Minimal — most read queries tracked unnecessarily |
| **dotnet-artisan** | ~65 | Comprehensive across all services |
| **dotnet-webapi** | ~75 | Most thorough coverage |
| **managedcode-dotnet-skills** | ~55 | Consistent across services |

**Verdict:** All skill configurations correctly use `AsNoTracking()` for read-only queries. The baseline's omission doubles memory usage and slows queries — a significant performance gap for list pages with eager loading.

## 8. EF Core Relationship Configuration

Only **managedcode-dotnet-skills** separates entity configuration into dedicated classes.

```csharp
// managedcode-dotnet-skills — IEntityTypeConfiguration<T> in Data/Configurations/
public class EventConfiguration : IEntityTypeConfiguration<Event>
{
    public void Configure(EntityTypeBuilder<Event> builder)
    {
        builder.Property(e => e.Status)
            .HasConversion<string>()
            .HasMaxLength(20);
        builder.HasOne(e => e.Category)
            .WithMany(c => c.Events)
            .HasForeignKey(e => e.EventCategoryId)
            .OnDelete(DeleteBehavior.Restrict);
    }
}

// DbContext just applies from assembly
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.ApplyConfigurationsFromAssembly(Assembly.GetExecutingAssembly());
}
```

```csharp
// no-skills / dotnet-artisan / dotnet-webapi — all inline in OnModelCreating
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Event>(entity =>
    {
        entity.HasOne(e => e.EventCategory)
            .WithMany(c => c.Events)
            .OnDelete(DeleteBehavior.Restrict);
        entity.Property(e => e.Status).HasConversion<string>();
    });
    // ... 100+ lines of inline configuration
}
```

All configurations consistently use:
- `OnDelete(DeleteBehavior.Restrict)` for critical relationships
- `HasConversion<string>()` for all enum properties
- Explicit decimal column types for monetary values
- Unique indexes on natural keys

**Verdict:** **managedcode-dotnet-skills** follows the cleanest pattern. Separate `IEntityTypeConfiguration<T>` classes keep `OnModelCreating` small and each entity's configuration isolated and testable. The inline approach in other configs creates bloated DbContext files.

## 9. Exception Handling Strategy

Only **dotnet-webapi** implements structured exception handling middleware.

```csharp
// dotnet-webapi/SparkEvents/Middleware/GlobalExceptionHandler.cs
internal sealed class GlobalExceptionHandler(
    ILogger<GlobalExceptionHandler> logger) : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, Exception exception, CancellationToken cancellationToken)
    {
        logger.LogError(exception, "An unhandled exception occurred: {Message}", exception.Message);
        httpContext.Response.Redirect("/Error");
        return true;
    }
}

// Registration in Program.cs
builder.Services.AddExceptionHandler<GlobalExceptionHandler>();
builder.Services.AddProblemDetails();
```

Other configurations rely solely on the default `app.UseExceptionHandler("/Error")` without custom handling, logging, or exception-type mapping.

**Verdict:** **dotnet-webapi** correctly implements `IExceptionHandler` (the modern .NET 8+ pattern). The dedicated `Middleware/` folder follows the skill's file placement guidance. Other configs miss structured error logging in production.

```csharp

{

    // ... seed data
    await context.SaveChangesAsync();
}
```

```csharp
// no-skills / dotnet-webapi — sync seeding (blocks startup thread)

```

**Verdict:** **dotnet-artisan** and **managedcode-dotnet-skills** correctly use async seeding, preventing thread pool starvation during app startup. The sync pattern in no-skills and dotnet-webapi blocks the startup thread unnecessarily.

## Summary: Impact of Skills

### Configuration Ranking (Overall Code Quality)

| Rank | Configuration | Score | Key Strengths | Key Gaps |
|---|---|---|---|---|
| **1** | **dotnet-webapi** | ⭐⭐⭐⭐⭐ | CancellationToken propagation, sealed types, IExceptionHandler, thin page models, AsNoTracking | Inline EF config, sync seeder, no custom tag helpers |
| **2** | **dotnet-artisan** | ⭐⭐⭐⭐½ | Sealed types, best collection expressions, custom TagHelper, async seeder, best accessibility/ARIA, breadcrumbs | No CancellationToken, hybrid page models (DbContext leaks into pages) |
| **3** | **managedcode-dotnet-skills** | ⭐⭐⭐⭐ | IEntityTypeConfiguration separation, async seeder, thin page models, ViewComponents, primary constructors | No sealed types, no CancellationToken, no custom exception handling |
| **4** | **no-skills** | ⭐⭐⭐ | Complete packages, solid baseline | Fat page models, no sealed types, traditional constructors, minimal AsNoTracking, no CancellationToken |

### Most Impactful Differences (Ranked by Production Impact)

1. **CancellationToken Propagation** (dotnet-webapi only) — Prevents wasted server resources on cancelled requests. **Critical for production** under load.

2. **AsNoTracking on Read Queries** (all skill configs) — 2 usages (no-skills) vs 55–75 (skill configs). Doubles memory efficiency on every list page.

3. **Sealed Types** (dotnet-artisan, dotnet-webapi) — Enables JIT devirtualization, communicates design intent. ~98% coverage vs 0%.

4. **IExceptionHandler Middleware** (dotnet-webapi only) — Structured exception logging and consistent error responses for production debugging.

5. **IEntityTypeConfiguration Separation** (managedcode only) — Prevents 100+ line DbContext bloat; each entity's config is isolated and testable.

6. **Primary Constructors** (all skill configs) — Eliminates ~5 lines of boilerplate per DI class. Unanimous adoption by skill configurations.

7. **Custom TagHelper** (dotnet-artisan) — Most Razor-idiomatic reusable UI component; produces clean `<status-badge>` syntax.

8. **Async Data Seeding** (dotnet-artisan, managedcode) — Prevents thread pool starvation during startup.

### Overall Assessment

**Skills make a measurable difference.** The baseline (no-skills) produces functional code that follows core .NET patterns, but all three skill configurations improve upon it in significant ways. The improvements are not cosmetic — CancellationToken propagation, AsNoTracking, sealed types, and structured exception handling are production-quality concerns that directly affect performance, scalability, and debuggability.

No single skill configuration captures every best practice. The ideal output would combine:
- **dotnet-webapi's** CancellationToken propagation and IExceptionHandler
- **dotnet-artisan's** sealed types, collection expressions, custom TagHelper, and ARIA accessibility
- **managedcode-dotnet-skills'** IEntityTypeConfiguration separation and ViewComponents

The **dotnet-artisan** plugin chain and **dotnet-webapi** skill produce the highest-quality output overall, with dotnet-webapi slightly ahead due to CancellationToken support — the single most impactful production differentiator.
