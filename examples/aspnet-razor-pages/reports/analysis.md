# Comparative Analysis: no-skills, dotnet-webapi, dotnet-artisan, managedcode-dotnet-skills

## Introduction

This report compares four ASP.NET Core Razor Pages application outputs generated by GitHub Copilot under different skill configurations:

| Configuration | Description |
|---|---|
| **no-skills** | Baseline — no Copilot skills attached |
| **dotnet-webapi** | Single `dotnet-webapi` skill providing .NET Web API best practices |
| **dotnet-artisan** | Plugin chain — 9 skills + 14 specialist agents for .NET development |
| **managedcode-dotnet-skills** | Community `managed-code` skills for .NET |

Each configuration generated the same three apps on .NET 10 with EF Core and SQLite:

- **SparkEvents** — Event registration portal (ticket types, capacity, waitlists, check-in)
- **KeystoneProperties** — Property management (leases, tenants, maintenance, rent tracking)
- **HorizonHR** — HR portal (employees, leave management, reviews, skills)

All four configurations produced fully functional, buildable applications with service-layer architecture, Bootstrap 5 UI, and comprehensive data seeding. The differences — detailed below across 30 dimensions — reveal how skills shape code quality, modern C# adoption, and production-readiness.

## 1. Page Model Design

**no-skills** — Page models inject both services *and* DbContext directly. Lookup queries are made directly against `_db` in page models, breaking the thin controller pattern:

```csharp
// no-skills: Events/Create.cshtml.cs — fat page model
public class CreateModel : PageModel
{
    private readonly IEventService _eventService;
    private readonly SparkEventsDbContext _db;
    // ...
    private async Task LoadLookupsAsync()
    {
        Categories = await _db.EventCategories.OrderBy(c => c.Name).ToListAsync();
        Venues = await _db.Venues.OrderBy(v => v.Name).ToListAsync();
    }
}
```

**dotnet-webapi** — Thin page models inject only service interfaces. All data access goes through services:

```csharp
// dotnet-webapi: Events/Create.cshtml.cs — thin page model
public sealed class CreateModel : PageModel
{
    private readonly IEventService _eventService;
    private readonly IEventCategoryService _categoryService;
    private readonly IVenueService _venueService;
    // No DbContext injection — all queries via services
}
```

**dotnet-artisan** — Mixed: uses primary constructors but still injects DbContext alongside services for dropdown lookups:

```csharp
// dotnet-artisan: Events/Create.cshtml.cs — leaks DbContext
public sealed class CreateModel(IEventService eventService, SparkEventsDbContext db) : PageModel
{
    private async Task LoadDropdownsAsync()
    {
        Categories = await db.EventCategories.OrderBy(c => c.Name)
            .Select(c => new SelectListItem(c.Name, c.Id.ToString())).ToListAsync();
    }
}
```

**managedcode** — Thin page models inject only services. Clean separation:

```csharp
// managedcode: Events/Create.cshtml.cs — thin page model
public class CreateModel(
    IEventService eventService,
    ICategoryService categoryService,
    IVenueService venueService) : PageModel
{
    private async Task LoadLookupsAsync()
    {
        var categories = await categoryService.GetAllAsync();
        CategoryList = new SelectList(categories, "Id", "Name");
    }
}
```

**Verdict:** **dotnet-webapi** and **managedcode** are best — fully thin page models with no DbContext leakage. dotnet-artisan and no-skills allow DbContext into page models, which reduces testability and violates single responsibility.

## 2. Form Handling & Validation

All four configurations use `[BindProperty]` with `InputModel` classes, `asp-validation-for` / `asp-validation-summary`, `_ValidationScriptsPartial` inclusion, and the PRG pattern (all return `RedirectToPage` from `OnPost`). None uses `[ValidateAntiForgeryToken]` explicitly — this is fine because Razor Pages applies it automatically via `AutoValidateAntiforgeryTokenAttribute`.

Key difference — **dotnet-webapi** adds `<label>` elements associated with inputs via `for` and `id` attributes on filter forms, while **no-skills** omits labels on filter inputs:

```html
<!-- dotnet-webapi: proper label association -->
<label for="Search" class="form-label">Search</label>
<input type="text" class="form-control" id="Search" name="search" ... />

<!-- no-skills: no labels on filter inputs -->
<input type="text" name="search" class="form-control" placeholder="Search by title..." />
```

**managedcode** uses `asp-for` tag helpers on filter forms (strongest approach):

```html
<!-- managedcode: tag helper binding on filters -->
<label class="form-label" asp-for="Search">Search</label>
<input asp-for="Search" class="form-control" placeholder="Search events..." />
```

**Verdict:** **managedcode** is best — uses `asp-for` tag helpers even on GET filter forms, ensuring model binding consistency. All configs correctly rely on Razor Pages' automatic anti-forgery.

### Key Takeaways

1. **dotnet-webapi** produces the most production-ready code. Its CancellationToken propagation (458 refs vs ≤6) and universal sealed types (188 vs 0) represent the largest quality gaps — these are patterns that are easy to miss without skill guidance but critical for production performance and correctness.

2. **dotnet-artisan** and **managedcode** each bring unique strengths that dotnet-webapi misses: artisan contributes custom TagHelpers and modern C# syntax patterns; managedcode contributes `IEntityTypeConfiguration`, ViewComponents, and the most comprehensive semantic HTML.

3. **no-skills** demonstrates that Copilot's baseline produces functional, well-structured applications with proper architecture — but misses modern C# patterns, performance optimizations, and production hardening that the skills provide.

4. **The ideal configuration would combine**: dotnet-webapi's CancellationToken/sealed/AsNoTracking discipline + managedcode's IEntityTypeConfiguration/ViewComponents + dotnet-artisan's TagHelpers/collection expressions. No single skill covers all best practices.
