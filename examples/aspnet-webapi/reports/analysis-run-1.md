# Comparative Analysis: dotnet-artisan, dotnet-skills, dotnet-webapi, managedcode-dotnet-skills, no-skills

## Introduction

This report compares ASP.NET Core Web API code generated by **five different Copilot skill configurations** across three identical application scenarios:

| Configuration | Description | Apps Generated (run-1) |
|---|---|---|
| **dotnet-artisan** | dotnet-artisan plugin chain | 3/3 (Fitness, Library, VetClinic) |
| **dotnet-skills** | Official .NET Skills (dotnet/skills) | 3/3 (Fitness, Library, VetClinic) |
| **dotnet-webapi** | dotnet-webapi skill | **1/3** (Fitness only) |
| **managedcode-dotnet-skills** | Community managed-code skills | 3/3 (Fitness, Library, VetClinic) |
| **no-skills** | Baseline (default Copilot) | 3/3 (Fitness, Library, VetClinic) |

Each app targets **.NET 10** with **EF Core + SQLite**, implementing a domain-rich REST API with business rules, seed data, and error handling. The three scenarios are:

- **FitnessStudioApi** — Booking/membership system with class scheduling, waitlists, and instructor management
- **LibraryApi** — Book loans, reservations, overdue fines, and availability tracking
- **VetClinicApi** — Pet healthcare with appointments, vaccinations, and medical records

> **Note:** The `dotnet-webapi` configuration only generated FitnessStudioApi in run-1. LibraryApi had only a `Directory.Build.props` (no source code) and VetClinicApi was entirely absent. Scores for dotnet-webapi are based on the single available app; the missing apps are penalized in completeness-sensitive dimensions.

## Summary: Impact of Skills

### Most Impactful Differences (ranked by weighted score impact)

1. **Minimal API Architecture** (15-point swing between best and worst) — The single largest differentiator. dotnet-artisan and dotnet-webapi produce modern Minimal APIs with `MapGroup`, `TypedResults`, and endpoint extension methods. managedcode and no-skills produce legacy Controller-based code. This is a fundamental architectural difference that affects code conciseness, performance, and OpenAPI generation.

2. **NuGet Package Discipline** (12-point swing) — dotnet-webapi pins all versions exactly; no-skills uses floating versions everywhere. Floating versions like `10.0.0-*` create non-reproducible builds and can silently introduce vulnerable packages.

3. **Modern C# Adoption** (10-point swing) — Primary constructors, collection expressions, sealed records, and the `required` keyword are consistently applied by dotnet-artisan and dotnet-webapi but partially or entirely absent in other configurations.

4. **Async & Cancellation Patterns** (10-point swing) — `CancellationToken` propagation through all layers is present in dotnet-artisan, dotnet-webapi, and partially in managedcode, but completely absent in no-skills and dotnet-skills.

5. **EF Core Read Performance** (10-point swing) — `AsNoTracking()` on read queries doubles performance and is consistently applied by dotnet-artisan and dotnet-webapi but ignored by no-skills.

### Overall Assessment

**dotnet-artisan** is the clear winner, combining the highest code quality with reliable generation of all three apps. It produces the most modern, production-ready code with Minimal APIs, sealed types, primary constructors, `CancellationToken` propagation, and is the only configuration that partially uses EF Core migrations.

**dotnet-webapi** produces the *highest quality individual code* — when it generates at all. Its FitnessStudioApi is a textbook example of modern .NET best practices. However, generating only 1 out of 3 apps makes it unreliable for real-world use. If the generation reliability issue were fixed, it would likely surpass dotnet-artisan.

**dotnet-skills** and **managedcode** occupy a middle tier. dotnet-skills is inconsistent — different apps within the same run follow different patterns (Controllers vs Minimal API, sealed vs unsealed, FluentValidation vs DataAnnotations). managedcode is more consistent but stuck on Controllers and Swashbuckle.

**no-skills** (baseline) produces functional code but with notably legacy patterns: no `CancellationToken`, no `AsNoTracking()`, no sealed types, middleware instead of `IExceptionHandler`, floating package versions, and Controllers instead of Minimal APIs. This clearly demonstrates the value that skills add to code generation quality.

The **universal blind spot** across all configurations is **security configuration** — no configuration implements HSTS or HTTPS redirection, and most use `EnsureCreated()` instead of migrations.
