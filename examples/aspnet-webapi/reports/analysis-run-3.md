# Comparative Analysis: no-skills, dotnet-artisan, managedcode-dotnet-skills, dotnet-skills, dotnet-webapi

## Introduction

This report analyzes code generated by **5 different Copilot skill configurations** across **3 identical ASP.NET Core Web API scenarios**. Each configuration generated implementations for:

- **FitnessStudioApi** — Booking/membership system with class scheduling, waitlists, and instructor management
- **LibraryApi** — Book loans, reservations, overdue fines, and availability tracking
- **VetClinicApi** — Pet healthcare with appointments, vaccinations, and medical records

| Configuration | Directory | Run | Description |
|---|---|---|---|
| **no-skills** | `output/no-skills/run-3/` | run-3 | Baseline default Copilot — no custom skills |
| **dotnet-artisan** | `output/dotnet-artisan/` | root | dotnet-artisan plugin chain |
| **managedcode-dotnet-skills** | `output/managedcode-dotnet-skills/` | root | Community managed-code skills |
| **dotnet-skills** | `output/dotnet-skills/` | root | Official .NET Skills (dotnet/skills) |
| **dotnet-webapi** | `output/dotnet-webapi/run-3/` | run-3 | dotnet-webapi custom skill |

> **Note**: Only `no-skills` and `dotnet-webapi` have `run-3` subdirectories. The remaining three configurations use root-level app directories as they lack numbered run folders for run-3.

All projects target **.NET 10** with **Entity Framework Core** and **SQLite**, and include seed data, `.http` test files, and OpenAPI documentation.

**no-skills** — Uses wildcard versions and outdated Swashbuckle:

## 3. Minimal API Architecture [CRITICAL]

This is the single biggest architectural differentiator across configurations.

**dotnet-webapi** — Full Minimal API with MapGroup, TypedResults, union return types, and endpoint extension methods in all 3 apps:
```csharp
// Program.cs (dotnet-webapi FitnessStudioApi)
app.MapMembershipPlanEndpoints();
app.MapMemberEndpoints();
app.MapBookingEndpoints();

// Endpoints/MemberEndpoints.cs (dotnet-webapi)
var group = app.MapGroup("/api/members").WithTags("Members");

group.MapGet("/", async Task<Ok<PaginatedResponse<MemberListResponse>>> (
    string? search, bool? isActive, int? page, int? pageSize,
    IMemberService service, CancellationToken ct) =>
{
    var result = await service.GetAllAsync(search, isActive,
        page ?? 1, Math.Clamp(pageSize ?? 20, 1, 100), ct);
    return TypedResults.Ok(result);
})
```

**dotnet-artisan** — Minimal API in FitnessStudioApi and VetClinicApi (2/3 apps), controllers in LibraryApi:
```csharp
// Endpoints/MembershipPlanEndpoints.cs (dotnet-artisan FitnessStudioApi)
public static RouteGroupBuilder MapMembershipPlanEndpoints(this IEndpointRouteBuilder routes)
{
    var group = routes.MapGroup("/api/membership-plans")
        .WithTags("Membership Plans");

    group.MapGet("/", async (IMembershipPlanService service, CancellationToken ct) =>
        TypedResults.Ok(await service.GetAllActiveAsync(ct)))
        .WithSummary("List all active membership plans");
```

**dotnet-skills** — Controllers in FitnessStudioApi and LibraryApi, inline Minimal API in VetClinicApi (endpoints defined directly in Program.cs without extension methods, using `Results.Ok()` not `TypedResults`):
```csharp
// Program.cs (dotnet-skills VetClinicApi) — inline endpoints, no extension methods
var owners = app.MapGroup("/api/owners").WithTags("Owners");
owners.MapGet("/", async (IOwnerService service, string? search, int? page, int? pageSize) =>
{
    return Results.Ok(await service.GetAllAsync(search, pagination));
}).WithSummary("List all owners");
```

**no-skills** and **managedcode** — Controllers everywhere:
```csharp
// MembersController.cs (no-skills FitnessStudioApi)
[ApiController]
[Route("api/members")]
[Produces("application/json")]
public class MembersController : ControllerBase
{
    private readonly IMemberService _service;
    public MembersController(IMemberService service) => _service = service;
}
```

| Config | Score | Justification |
|---|---|---|
| no-skills | 1 | Controllers in all 3 apps — no Minimal API patterns |
| dotnet-artisan | 4 | Minimal API with MapGroup/TypedResults in 2/3 apps |
| managedcode | 1 | Controllers in all 3 apps — no Minimal API patterns |
| dotnet-skills | 2 | Controllers in 2/3, inline Minimal in Vet without TypedResults or extension methods |
| dotnet-webapi | 5 | Full Minimal API in all 3 apps with MapGroup, TypedResults, union return types, extension methods |

**Verdict**: **dotnet-webapi** is the clear winner. Minimal APIs with route groups and TypedResults are the modern .NET standard — they produce more concise code and provide compile-time type safety with automatic OpenAPI schema generation. **dotnet-artisan** is a strong second.

## 4. Input Validation & Guard Clauses [CRITICAL]

All configurations use Data Annotations on DTOs. No configuration uses `ArgumentNullException.ThrowIfNull()` for guard clauses.

**dotnet-webapi** — Consistent validation with `required` keyword and `Range` attributes:
```csharp
// DTOs/MemberDtos.cs (dotnet-webapi FitnessStudioApi)

{
    [Required, MaxLength(100)]
    public required string FirstName { get; init; }

    [Required, EmailAddress]
    public required string Email { get; init; }

    [Required, MaxLength(200)]
    public required string EmergencyContactName { get; init; }
}
```

**dotnet-skills** — FluentValidation in FitnessStudioApi (3rd-party dependency):
```csharp
// Validators.cs (dotnet-skills FitnessStudioApi)
public sealed class CreateMemberValidator : AbstractValidator<CreateMemberDto>
{
    public CreateMemberValidator()
    {
        RuleFor(x => x.FirstName).NotEmpty().MaximumLength(100);
        RuleFor(x => x.Email).NotEmpty().EmailAddress();
        RuleFor(x => x.DateOfBirth).Must(dob => {
            var age = /* age calc */;
            return age >= 16;
        }).WithMessage("Member must be at least 16 years old.");
    }
}
```

**no-skills** — Standard Data Annotations:
```csharp
// Dtos.cs (no-skills FitnessStudioApi)

{

    [Required, MaxLength(100)]
    public string FirstName { get; set; } = string.Empty;

```

All configs validate business rules in services with exception throwing, but none use `ArgumentNullException.ThrowIfNull()` or `ArgumentException.ThrowIfNullOrEmpty()`.

| Config | Score | Justification |
|---|---|---|
| no-skills | 3 | Data annotations, business validation via exceptions, no guard clause helpers |
| dotnet-artisan | 3 | Data annotations, guard clauses with exceptions, no ThrowIfNull |
| managedcode | 3 | Data annotations + FluentValidation in Library, guard clauses |
| dotnet-skills | 3 | Data annotations + FluentValidation in Fitness, guard clauses |
| dotnet-webapi | 4 | Data annotations with `required` keyword, `Range` validation, consistent guard patterns |

**Verdict**: **dotnet-webapi** edges ahead with `required` properties enforcing non-null at compile time. All configs could improve by adding `ArgumentNullException.ThrowIfNull()` on DI constructor parameters.

## 6. EF Migration Usage [CRITICAL]

This is a stark differentiator — only **dotnet-webapi** uses migrations.

**dotnet-webapi** — Uses `MigrateAsync()` across all 3 apps:
```csharp
// Program.cs (dotnet-webapi FitnessStudioApi)
using (var scope = app.Services.CreateScope())
{
    var db = scope.ServiceProvider.GetRequiredService<FitnessStudioDbContext>();
    await db.Database.MigrateAsync();
    await DataSeeder.SeedAsync(db);
}
```

**All other configs** — Use `EnsureCreated()` or `EnsureCreatedAsync()`:
```csharp
// Program.cs (no-skills FitnessStudioApi)
var db = scope.ServiceProvider.GetRequiredService<FitnessDbContext>();
await db.Database.EnsureCreatedAsync();
await seeder.SeedAsync();

// Program.cs (dotnet-artisan FitnessStudioApi)
await db.Database.EnsureCreatedAsync();
await DataSeeder.SeedAsync(db);

// Program.cs (dotnet-skills FitnessStudioApi)
context.Database.EnsureCreated();
DataSeeder.Seed(context);

// Program.cs (managedcode FitnessStudioApi)
await db.Database.EnsureCreatedAsync();
await DataSeeder.SeedAsync(db);
```

| Config | Score | Justification |
|---|---|---|
| no-skills | 1 | EnsureCreated in all 3 apps |
| dotnet-artisan | 1 | EnsureCreated in all 3 apps |
| managedcode | 1 | EnsureCreated in all 3 apps |
| dotnet-skills | 1 | EnsureCreated in all 3 apps |
| dotnet-webapi | 5 | MigrateAsync in all 3 apps |

**Verdict**: **dotnet-webapi** is the only configuration using the production-safe migration pattern. `EnsureCreated()` bypasses EF Core migrations, making schema evolution impossible and risking data loss on model changes.

## 10. Error Handling & Middleware [HIGH]

The modern approach is `IExceptionHandler` (introduced in .NET 8) with `AddProblemDetails()`.

**dotnet-webapi** — Uses `IExceptionHandler` with switch expressions:
```csharp
// Middleware/ApiExceptionHandler.cs (dotnet-webapi FitnessStudioApi)
internal sealed class ApiExceptionHandler(ILogger<ApiExceptionHandler> logger)
    : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, Exception exception, CancellationToken cancellationToken)
    {
        var (statusCode, title) = exception switch
        {
            KeyNotFoundException => (StatusCodes.Status404NotFound, "Not Found"),
            ArgumentException => (StatusCodes.Status400BadRequest, "Bad Request"),
            InvalidOperationException => (StatusCodes.Status409Conflict, "Conflict"),
            _ => (0, (string?)null)
        };
        // ...
        await httpContext.Response.WriteAsJsonAsync(new ProblemDetails { ... }, cancellationToken);
        return true;
    }
}

// Program.cs
builder.Services.AddExceptionHandler<ApiExceptionHandler>();
builder.Services.AddProblemDetails();
app.UseExceptionHandler();
app.UseStatusCodePages();
```

```csharp
// GlobalExceptionHandlerMiddleware.cs (no-skills FitnessStudioApi)
public class GlobalExceptionHandlerMiddleware
{

    public async Task InvokeAsync(HttpContext context)
    {
        try { await _next(context); }
        catch (Exception ex)
        {
            var problemDetails = new ProblemDetails { ... };
            await context.Response.WriteAsync(json);
        }
    }
}
```

## 12. EF Core Best Practices [HIGH]

```csharp
entity.Property(e => e.Status).HasConversion<string>();
```

### Most Impactful Differences

1. **EF Migration vs EnsureCreated** (Critical, 4-point gap): Only dotnet-webapi uses `MigrateAsync()`. This is the single most impactful architectural decision — EnsureCreated is a data loss risk in any real deployment.

2. **Minimal API Architecture** (Critical, 4-point gap): dotnet-webapi consistently generates Minimal APIs with `MapGroup()`, `TypedResults`, and `Results<T1, T2>` union types. No-skills and managedcode always produce controller-based code — the legacy pattern.

3. **NuGet Version Pinning** (Critical, 3-point gap): dotnet-webapi pins exact versions (`10.0.4`); all others use wildcards (`10.*`, `10.0.0-*`). Wildcard versions are the most fundamental reproducibility violation.

4. **Modern C# Adoption** (High, 3-point gap): dotnet-webapi and dotnet-artisan use primary constructors, sealed types, collection expressions, and `required` properties. No-skills generates pre-C# 12 patterns.

5. **CancellationToken Propagation** (High, 3-point gap): dotnet-webapi propagates tokens from endpoints through services to EF Core calls. No-skills completely omits `CancellationToken`.

### Overall Assessment

**dotnet-webapi** is the clear winner, scoring 73% higher than the no-skills baseline (189.5 vs 109.5). Its advantages are concentrated in Critical dimensions — the areas that matter most for production-quality .NET APIs. The combination of Minimal APIs, EF migrations, exact version pinning, and complete CancellationToken propagation makes it the only configuration that consistently produces code meeting modern .NET team standards.

**dotnet-artisan** is a strong second place, sharing many of dotnet-webapi's modern C# patterns (primary constructors, sealed types, collection expressions) but falling short on EF migrations (still uses EnsureCreated) and package version discipline (wildcards).

**managedcode-dotnet-skills** and **dotnet-skills** occupy the middle ground. Both improve on the baseline with IExceptionHandler and CancellationToken, but neither achieves Minimal API architecture consistently. The managedcode config edges ahead with more consistent modern patterns.

**no-skills** (baseline) generates functional but architecturally dated code. The absence of primary constructors, sealed types, CancellationToken, AsNoTracking, and Minimal APIs means the generated code requires significant modernization before it meets current .NET best practices.

The single biggest gap between the best and worst configurations is in **Critical dimensions** — exactly where quality matters most.
