# Comparative Analysis: no-skills, dotnet-webapi, dotnet-artisan, managedcode-dotnet-skills, dotnet-skills

## Introduction

This report compares code generated by **five different Copilot configurations** across **three ASP.NET Core Web API scenarios**, totaling **15 independent implementations**. Each configuration was given the same prompts to generate:

| Scenario | Domain |
|----------|--------|
| **FitnessStudioApi** | Booking/membership system with class scheduling, waitlists, and instructor management |
| **LibraryApi** | Book loans, reservations, overdue fines, and availability tracking |
| **VetClinicApi** | Pet healthcare with appointments, vaccinations, and medical records |

### Configurations

| Directory | Configuration | Description |
|-----------|--------------|-------------|
| `no-skills` | Baseline | Default Copilot with no custom skills |
| `dotnet-webapi` | dotnet-webapi skill | Single skill focused on modern .NET Web API patterns |
| `dotnet-artisan` | dotnet-artisan plugin chain | Multi-skill plugin (dotnet-advisor → dotnet-csharp → dotnet-api) |
| `managedcode-dotnet-skills` | Community managed-code skills | Community-maintained .NET skills (dotnet-aspnet-core, dotnet-entity-framework-core, dotnet-modern-csharp, etc.) |
| `dotnet-skills` | Official .NET Skills (dotnet/skills) | Microsoft's official dotnet/skills (analyzing-dotnet-performance, optimizing-ef-core-queries) |

All projects target **.NET 10** with **Entity Framework Core** and **SQLite**.

## 1. API Style

### What each configuration does

**no-skills** — Uses traditional MVC controllers with `[ApiController]` across all three apps:

```csharp
// no-skills: OwnersController.cs
[ApiController]
[Route("api/[controller]")]
public class OwnersController : ControllerBase
{
    private readonly IOwnerService _service;
    public OwnersController(IOwnerService service) { _service = service; }

    [HttpGet]
    public async Task<IActionResult> GetAll([FromQuery] string? search, ...) { ... }
}
```

**dotnet-webapi** — Uses Minimal APIs with `MapGroup()` and static extension methods across all three apps:

```csharp
// dotnet-webapi: OwnerEndpoints.cs
public static class OwnerEndpoints
{
    public static void MapOwnerEndpoints(this IEndpointRouteBuilder app)
    {
        var group = app.MapGroup("/api/owners").WithTags("Owners");
        group.MapGet("/", async Task<Results<Ok<PaginatedResponse<OwnerResponse>>, BadRequest>> (
            string? search, int? page, int? pageSize,
            IOwnerService service, CancellationToken ct) => { ... });
    }
}
```

**dotnet-artisan** — Mixed: Minimal APIs for FitnessStudioApi and VetClinicApi (14 endpoint files), but Controllers for LibraryApi (7 controller files). The gen-notes acknowledge this trade-off.

**managedcode-dotnet-skills** — Controllers across all three apps (21 controller files, 0 endpoint files).

**dotnet-skills** — Controllers across all three apps (14 controller files, 0 endpoint files), but VetClinicApi uses inline Minimal API endpoint lambdas directly in Program.cs with `MapGroup()` (7 groups).

**Verdict:** **dotnet-webapi wins.** Minimal APIs are the modern .NET default, have lower overhead, produce more concise code, and enable `TypedResults` for automatic OpenAPI generation. The `no-skills` and `managedcode-dotnet-skills` configurations consistently default to the older controller pattern.

## 2. Sealed Types

### What each configuration does

**no-skills** — Zero `sealed` declarations across all files. All classes are open for inheritance:

```csharp
// no-skills: BookingService.cs
public class BookingService : IBookingService { ... }
```

**dotnet-webapi** — 126 `sealed` declarations. All services, DTOs, middleware, entities, and the DbContext are sealed:

```csharp
// dotnet-webapi: BookingService.cs
public sealed class BookingService(FitnessDbContext db, ILogger<BookingService> logger)
    : IBookingService { ... }
```

**dotnet-artisan** — 132 `sealed` declarations. Consistent sealing across all types:

```csharp
// dotnet-artisan: Owner.cs
public sealed class Owner { ... }
```

**managedcode-dotnet-skills** — Zero `sealed` declarations despite using primary constructors.

**dotnet-skills** — 68 `sealed` declarations, applied after initial generation by the `analyzing-dotnet-performance` skill pass.

**Verdict:** **dotnet-artisan wins** (132 sealed), closely followed by **dotnet-webapi** (126). Sealing types satisfies CA1852 and enables JIT devirtualization optimizations. The `no-skills` and `managedcode-dotnet-skills` configurations miss this entirely.

## 4. DTO Design

### What each configuration does

**no-skills** — Positional records but unsealed, using `*Dto` naming and `IEnumerable<T>` in pagination:

```csharp
// no-skills: Dtos.cs
public record BookingDto(int Id, int ClassScheduleId, string ClassName, ...);
public record CreateBookingDto(int ClassScheduleId, int MemberId);
public record PaginatedResult<T>(IEnumerable<T> Items, int TotalCount, int Page, int PageSize, int TotalPages);
```

**dotnet-webapi** — Sealed records with `*Request`/`*Response` naming, `required init` properties on request DTOs:

```csharp
// dotnet-webapi: BookingDtos.cs
public sealed record CreateBookingRequest
{
    [Required]
    public required int ClassScheduleId { get; init; }
    [Required]
    public required int MemberId { get; init; }
}

public sealed record BookingResponse(int Id, int ClassScheduleId, string ClassName, ...);
```

**dotnet-artisan** — Sealed records with `*Request`/`*Response` naming, same pattern as dotnet-webapi:

```csharp
// dotnet-artisan: BookingDtos.cs
public sealed record BookingResponse(int Id, int ClassScheduleId, string ClassTypeName, ...);
public sealed record CreateBookingRequest { public int ClassScheduleId { get; init; } }
```

**managedcode-dotnet-skills** — Records but unsealed, using `*Dto` naming:

```csharp
// managedcode-dotnet-skills: BookingDtos.cs
public record BookingDto(int Id, int ClassScheduleId, string ClassTypeName, ...);
public record CreateBookingDto { [Required] public int ClassScheduleId { get; init; } }
```

**dotnet-skills** — Mutable **classes** (not records) with `get; set;` properties:

```csharp
// dotnet-skills: BookingDtos.cs
public sealed class BookingDto
{
    public int Id { get; set; }
    public string ClassName { get; set; } = string.Empty;
    public DateTime BookingDate { get; set; }
}

public sealed class CreateBookingDto
{
    public int ClassScheduleId { get; set; }
    public int MemberId { get; set; }
}
```

**Verdict:** **dotnet-webapi wins.** Sealed records with `required init` properties provide immutability, value equality, and clear API contracts. The `*Request`/`*Response` naming convention is more expressive than `*Dto`. The **dotnet-skills** approach of mutable DTO classes is the weakest — losing both immutability and value semantics.

## 5. Service Abstraction

### What each configuration does

All five configurations use **interface-based service abstraction** with `AddScoped<IService, Service>()`:

```csharp
// All configurations: Program.cs
builder.Services.AddScoped<IOwnerService, OwnerService>();
builder.Services.AddScoped<IPetService, PetService>();
builder.Services.AddScoped<IVeterinarianService, VeterinarianService>();
```

The difference is in interface **organization**:

| Configuration | Interface Location |
|---|---|
| no-skills | Single `Interfaces.cs` / `IServices.cs` file |
| dotnet-webapi | Individual `IOwnerService.cs`, `IPetService.cs` files |
| dotnet-artisan | Individual files per interface |
| managedcode-dotnet-skills | Mixed — some single `Interfaces.cs`, some individual |
| dotnet-skills | Subdirectory `Services/Interfaces/` with individual files |

**Verdict:** **dotnet-webapi** and **dotnet-artisan** follow best practice with one-interface-per-file. **dotnet-skills** takes it further with a dedicated `Interfaces/` subdirectory which is a reasonable alternative.

## 6. CancellationToken Propagation

### What each configuration does

**no-skills** — Zero `CancellationToken` usages. No cancellation support at all:

```csharp
// no-skills: OwnersController.cs
public async Task<IActionResult> GetAll([FromQuery] string? search, ...)
{
    var result = await _service.GetAllAsync(search, ...);  // no CancellationToken
    return Ok(result);
}
```

**dotnet-webapi** — 354 usages. Full propagation from endpoints through services to EF Core:

```csharp
// dotnet-webapi: OwnerEndpoints.cs
group.MapGet("/", async Task<Results<Ok<PaginatedResponse<OwnerResponse>>, BadRequest>> (
    string? search, int? page, int? pageSize,
    IOwnerService service, CancellationToken ct) =>  // ← injected
{
    var result = await service.GetAllAsync(search, p, ps, ct);  // ← forwarded
    return TypedResults.Ok(result);
});

// dotnet-webapi: OwnerService.cs
public async Task<PaginatedResponse<OwnerResponse>> GetAllAsync(
    string? search, int page, int pageSize, CancellationToken ct)
{
    var query = db.Owners.AsNoTracking();
    var totalCount = await query.CountAsync(ct);  // ← forwarded to EF Core
    var items = await query.Skip(...).Take(...).ToListAsync(ct);
}
```

**dotnet-artisan** — 233 usages with `ct = default` pattern for optional cancellation:

```csharp
// dotnet-artisan: OwnerEndpoints.cs
private static async Task<Ok<PaginatedResponse<OwnerDto>>> GetAll(
    IOwnerService service, ..., CancellationToken ct = default)
```

**managedcode-dotnet-skills** — 233 usages. Full propagation with `CancellationToken ct` or `ct = default`:

```csharp
// managedcode-dotnet-skills: OwnersController.cs
public async Task<IActionResult> GetAll(..., CancellationToken ct = default)
{
    var result = await ownerService.GetAllAsync(search, page, pageSize, ct);
}
```

**dotnet-skills** — Only 3 usages across all code. Virtually no cancellation support:

```csharp
// dotnet-skills: BookingService.cs
public async Task<BookingDto> CreateAsync(CreateBookingDto dto)  // no CancellationToken
{
    var member = await _context.Members.FindAsync(dto.MemberId);  // no ct
}
```

**Verdict:** **dotnet-webapi wins** with the most thorough propagation (354 usages). This is critical for production — without `CancellationToken`, the server continues processing requests even after the client disconnects, wasting resources. The **no-skills** and **dotnet-skills** configurations completely miss this.

## 7. AsNoTracking Usage

### What each configuration does

**no-skills** — Zero `AsNoTracking()` usages. All queries create tracked entity instances:

```csharp
// no-skills: OwnerService.cs — no AsNoTracking
var owner = await _db.Owners.Include(o => o.Pets).FirstOrDefaultAsync(o => o.Id == id);
```

**dotnet-webapi** — 75 usages. Consistently applied on all read-only queries:

```csharp
// dotnet-webapi: BookingService.cs
var booking = await db.Bookings
    .AsNoTracking()
    .Include(b => b.ClassSchedule)
        .ThenInclude(cs => cs.ClassType)
    .Include(b => b.Member)
    .FirstOrDefaultAsync(b => b.Id == id, ct);
```

**dotnet-artisan** — 59 usages. Consistent application on GET queries.

**managedcode-dotnet-skills** — 60 usages. Consistent usage across read-only operations.

**dotnet-skills** — 39 usages. Applied but less consistently than dotnet-webapi.

**Verdict:** **dotnet-webapi wins** (75 usages). `AsNoTracking()` has a major performance impact — tracking doubles memory usage and slows queries under load. The **no-skills** baseline completely misses this optimization.

## 9. Data Seeder Design

### What each configuration does

**no-skills** — Injectable service with `ILogger<T>` and idempotent check:

```csharp
// no-skills: DataSeeder.cs
public class DataSeeder
{
```

**managedcode-dotnet-skills** — Static `DataSeeder.SeedAsync(db)` with idempotent checks.

## 10. Middleware Style

### What each configuration does

**no-skills** — Convention-based middleware with `RequestDelegate`:

```csharp
// no-skills: GlobalExceptionHandlerMiddleware.cs
public class GlobalExceptionHandlerMiddleware
{
    private readonly RequestDelegate _next;
    public GlobalExceptionHandlerMiddleware(RequestDelegate next, ILogger<...> logger) { ... }
    public async Task InvokeAsync(HttpContext context)
    {
        try { await _next(context); }
        catch (BusinessRuleException ex) { ... }
        catch (Exception ex) { ... }
    }
}
// Registered as: app.UseMiddleware<GlobalExceptionHandlerMiddleware>();
```

**dotnet-webapi** — Modern `IExceptionHandler` with pattern matching:

```csharp
// dotnet-webapi: ApiExceptionHandler.cs
internal sealed class ApiExceptionHandler(ILogger<ApiExceptionHandler> logger)
    : IExceptionHandler
{
    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, Exception exception, CancellationToken cancellationToken)
    {
        var (statusCode, title) = exception switch
        {
            KeyNotFoundException => (StatusCodes.Status404NotFound, "Not Found"),
            ArgumentException => (StatusCodes.Status400BadRequest, "Bad Request"),
            InvalidOperationException => (StatusCodes.Status409Conflict, "Conflict"),
            _ => (0, (string?)null)
        };
        ...
    }
}
// Registered as: builder.Services.AddExceptionHandler<ApiExceptionHandler>();
```

**dotnet-artisan** — `IExceptionHandler` with primary constructor (all 3 apps).

**managedcode-dotnet-skills** — `IExceptionHandler` with custom exception hierarchy and switch expression (all 3 apps).

**dotnet-skills** — Mixed: 1 app uses `IExceptionHandler`, 2 use convention-based `RequestDelegate` middleware.

**Verdict:** **dotnet-webapi wins.** `IExceptionHandler` is the modern .NET 8+ recommended pattern — composable, DI-aware, supports `CancellationToken`, and integrates with `AddProblemDetails()`. Convention middleware requires manual `try/catch` wrapping and lacks composability.

## 11. Exception Handling Strategy

### What each configuration does

**no-skills** — Custom `BusinessRuleException` class with `StatusCode` and `Title` properties:

```csharp
// no-skills: GlobalExceptionHandlerMiddleware.cs
public class BusinessRuleException : Exception
{
    public int StatusCode { get; }
    public string Title { get; }
    public BusinessRuleException(string message, int statusCode = 400, string title = "Business Rule Violation") { ... }
}
```

**dotnet-webapi** — Uses **built-in .NET exception types** mapped to HTTP status codes in the handler:

```csharp
// dotnet-webapi: BookingService.cs
throw new KeyNotFoundException($"Member with ID {request.MemberId} not found.");  // → 404
throw new ArgumentException("Inactive members cannot book classes.");              // → 400
throw new InvalidOperationException("Cannot book that class.");                    // → 409
```

**dotnet-artisan** — Same built-in exception types as dotnet-webapi (`KeyNotFoundException`, `InvalidOperationException`).

**managedcode-dotnet-skills** — Custom exception **hierarchy** with typed exceptions:

```csharp
// managedcode-dotnet-skills: GlobalExceptionHandler.cs
public class BusinessRuleException(string message) : Exception(message);
public class ConflictException(string message) : Exception(message);
public class NotFoundException(string message) : Exception(message);
```

**dotnet-skills** — Custom `BusinessRuleException` with `StatusCode` and `Title`, plus `KeyNotFoundException`.

**Verdict:** **managedcode-dotnet-skills** has the cleanest exception hierarchy with semantic types. **dotnet-webapi** is pragmatic — using built-in types avoids custom exception proliferation. Both are valid approaches; the key differentiator is consistent mapping to HTTP status codes.

## 12. File Organization

### What each configuration does

All configurations follow a **per-concern folder structure** with minor variations:

| Configuration | Structure |

| dotnet-webapi | `Endpoints/`, `Services/`, `Models/`, `DTOs/`, `Data/`, `Middleware/` |

**Verdict:** All are reasonable. **dotnet-webapi** and **dotnet-artisan** are cleanest with flat per-concern folders. **dotnet-skills** over-structures with nested subdirectories that add navigational overhead for small projects.

## 13. Pagination

### What each configuration does

**no-skills** — Mutable class with `List<T>`:

```csharp
// no-skills: PagedResult.cs
public class PagedResult<T>
{
    public List<T> Items { get; set; } = new();
    public int TotalCount { get; set; }
    public int Page { get; set; }
    public int PageSize { get; set; }
    public int TotalPages => (int)Math.Ceiling((double)TotalCount / PageSize);
}
```

**dotnet-webapi** — Sealed record with `IReadOnlyList<T>`, computed properties, and factory method:

```csharp
// dotnet-webapi: PaginatedResponse.cs
public sealed record PaginatedResponse<T>
{
    public required IReadOnlyList<T> Items { get; init; }
    public required int Page { get; init; }
    public required int PageSize { get; init; }
    public required int TotalCount { get; init; }
    public required int TotalPages { get; init; }
    public required bool HasNextPage { get; init; }
    public required bool HasPreviousPage { get; init; }

    public static PaginatedResponse<T> Create(
        IReadOnlyList<T> items, int page, int pageSize, int totalCount) { ... }
}
```

**dotnet-artisan** — Sealed positional record with computed properties:

```csharp
// dotnet-artisan: PaginatedResponse.cs
public sealed record PaginatedResponse<T>(
    IReadOnlyList<T> Items, int TotalCount, int Page, int PageSize)
{
    public int TotalPages => (int)Math.Ceiling(TotalCount / (double)PageSize);
    public bool HasNextPage => Page < TotalPages;
    public bool HasPreviousPage => Page > 1;
}
```

**managedcode-dotnet-skills** — Unsealed record with computed properties (similar to artisan but not sealed):

```csharp
// managedcode-dotnet-skills: PaginatedResponse.cs

{
```
